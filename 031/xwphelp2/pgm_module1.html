De "Module" pagina's bevinden zich in de instellingen notitieblokken van alle uitvoerbare bestanden om extra informatie te laten zien van een uitvoerbare module.

Deze pagina toont u de submodules die door de module zijn ge‹mporteerd.

De pagina maakt alleen een lijst van de submodules die expliciet zijn genoteerd als ge‹mporteerd in de module header voor 'load-time linking'. Deze ge‹mporteerde submodules zijn reeds verdwenen op het moment dat de module geladen wordt, in tegenstelling tot 'run-time linking' (zie beneden). De namen van de modules zijn de namen van de dynamic link libraries die moeten worden ge‹mporteerd meestal zonder de .DLL bestandsnaam toevoeging, en worden verwacht in het LIBPATH (zie "Systeempaden" voor meer informatie).

Als een uitvoerbaar bestand niet kan worden geladen, kan de informatie op deze pagina handig zijn om uit te vinden welke submodule gefaald heeft.

Echter, als resultaat van het bovenstaande, laat deze pagina niet zien de:

  • Importeren tijdens Run-time, als een applicatie specifieke &os2; API's aanroept om code dynamisch te importeren nadat die compleet geladen is. Deze informatie wordt niet in de header van het uitvoerbare bestand gespecificeerd omdat het de actuele programma code is die het importeren uitvoert. Zie de documentatie over DosQueryProcAddr in de Toolkit documentatie. *

  • Indirecte importen, d.w.z., subimporten uitgevoerd door de modules genoteerd op deze pagina. Deze importen hangen af van de uitvoeringsinhoud: b.v., de werkelijke actieve LIBPATH, BEGINLIBPATH, en ENDLIBPATH systeem paden bepalen allemaal welke modules werkelijk zullen worden geladen.
Om te bepalen welke module echt wordt gebruikt door een bepaald programma, kunt U de PSTAT /L opdracht uitvoeren vanaf een &os2; opdrachtaanwijzing terwijl het specifieke programma draait.
